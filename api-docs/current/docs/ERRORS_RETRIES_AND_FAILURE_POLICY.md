# ClawPad API Errors, Retries, and Failure Policy

Version: 1.0

## 1. Response Envelope

All API responses are wrapped:

```json
{
  "success": true,
  "data": {},
  "error": null,
  "timestamp": "2026-02-21T00:00:00+00:00"
}
```

Failure example:

```json
{
  "success": false,
  "error": "Access token expired",
  "timestamp": "2026-02-21T00:00:00+00:00"
}
```

## 2. Status Code Matrix

### 2.1 `400 Bad Request`

Typical causes:

- Invalid required fields.
- Malformed headers or payload format.
- Invalid timestamp header type.

Action:

- Do not retry unchanged request.
- Fix request construction.

### 2.2 `401 Unauthorized`

Typical causes:

- Missing/invalid bearer token.
- Expired/revoked token.
- Signature mismatch.
- Replay nonce.
- Timestamp outside skew window.

Action:

1. Refresh token once if possible.
2. Re-sign and retry once.
3. If still failing, re-enroll client.

### 2.3 `403 Forbidden`

Typical causes:

- Scope mismatch (`admin`/`write` required).
- Enrollment bootstrap route called from non-localhost.

Action:

- Treat as policy/config issue. No blind retries.

### 2.4 `404 Not Found`

Typical causes:

- Missing project/file/client resource.

Action:

- Treat as domain-state failure. No automatic retries.

### 2.5 `5xx Server Errors`

Typical causes:

- Internal service exceptions.
- Temporary runtime dependency problems.

Action:

- Retry with exponential backoff.
- Cap retries and alert operators.

## 3. Retry Policy (Recommended)

- Network error: retry.
- `500-599`: retry.
- `429`: currently not part of default contract (no built-in global rate limiter documented).
- `401`: one refresh + one retry max.
- `400/403/404`: no automatic retries.

Backoff template:

- Base: 250 ms
- Factor: x2
- Jitter: 20%
- Max attempts: 4

## 4. No-Fallback Contract

ClawPad integrations should preserve explicit failure behavior:

- Do not fabricate default success payloads.
- Do not replace failed responses with synthetic placeholders.
- If data is unavailable, return or surface the real error.

## 5. Logging and Diagnostics

Include in structured logs:

- HTTP method and path.
- Status code.
- `error` message from envelope.
- `gateway_fingerprint` (when available).
- Correlation/request ID generated by client side.

## 6. Incident Playbook

1. Confirm `/health` status.
2. Confirm token freshness and clock drift.
3. Validate signature inputs (path/body hash/token hash).
4. Check nonce reuse in client code.
5. Escalate after bounded retries.
